Web application firewall
A Web Application Firewall (WAF) sits in front of web applications to filter out malicious traffic at the application layer (Layer 7 of the Open Systems Interconnection (OSI) Model, including HTTPS and HTTP traffic). A WAF stops attacks at the network edge, protecting your website from common web threats and specialized attacks before they reach your servers.

Content Delivery Network (CDN) allows placements of assets such as images and video, and even whole web pages, at the very edge of the network closet to customers to ensure top performance. DDoS mitigation
DDoS (Distributed Denial of Service) attacks are common and complex, and traditional on-premise solutions cannot handle these. Episerver DXC Service offers advanced protection at the network edge through its CDN provider including UDP and ICMP protocols, DNS amplification, Layer 7 and 3/4, SYN/ACK, and SMURF (refer to information on the net for this terminology). Microsoft Azure also protects against attacks generated from outside and inside the platform.

Database retention
Episerver performs regular backups of the application and database. Episerver saves backup copies for thirty-five (35) days after which Episerver does not take responsibility for the backup information (retention 35 days). Files stored in Azure Blob Storage are not backed up as a part of this process. Azure Blob Storage is disaster resilient as it is replicated both within the data center and to a geo-redundant location.

TLS/SSL is commonly used for encrypted integration and communication with other services over HTTP (HTTPS). All domains in the service are protected by TLS/SSL by default. Production scales automatically, and deployment is done by Episerver. See the DXC Service Environments section on Episerver World for additional information.

Preproduction
Episerver uses this environment to test production deployment, and verify performance and operational functionality. Customers may also use for UAT, load testing, or approved penetration testing. Preproduction scales automatically, and deployment is done by Episerver. Any content added in this environment will be overwritten during an upgrade. See the DXC Service Environments section on Episerver World for additional information.

Integration
Partners and customer deploy the full solution, as daily builds or continuous releases. Here customers can validate initial integrations with external systems, perform functional testing, and add initial content in the case of a first-time deployment. The integration environment has fixed configuration and no automatic scaling. You can deploy updates to Integration yourself. See the DXC Service Environments section on Episerver World for additional information.

Additional deployment environments (max)
This is an Integration-like environment that is a replica of the primary Integration environment. The partner can deploy directly just like they would to the other standard Integration environment. However, the primary Integration environment in the DXC Service is considered the ""master"" and is the only one that can be used to follow the standard deployment flow to Preproduction/Production. The customer can deploy directly to the Additional Integration environment. When they are ready to deploy to Preproduction/Production they must first deploy to the primary Integration environment. This Additional Integration environment can never be used to deploy to Preproduction/Production. Common usage would be when there are multiple development streams running in parallel.
